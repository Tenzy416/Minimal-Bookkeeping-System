## 1. 開發過程 (Development Process)

本系統依據 **軟體開發生命週期 (SDLC)** 的核心階段進行開發：
1. **規劃與需求分析**：針對大學生「理財困難、缺乏記帳習慣」的痛點，規劃出「極簡、無廣告、三秒記帳」的輕量化 Web App。定義了三大核心功能（新增收支、歷史清單、總額統計），並排除登入、圖表等次要功能以利初期聚焦。
2. **資料庫與系統設計**：採用輕量級關聯式資料庫 SQLite，設計單一 `expenses` 資料表，確保資料存取結構化；利用 Mermaid 繪製前後端分離的系統架構圖，確保資料傳遞流程清晰。
3. **功能開發 (前後端同步)**：
   * **前端**：使用原生 HTML / CSS / JavaScript，確保載入速度極快。利用 CSS Flexbox 與 Grid 進行佈局，並搭配 FontAwesome 圖標提升視覺質感。
   * **後端**：以 Python Flask 框架架設 RESTful API，並直接託管前端靜態網頁檔案，解決跨來源資源共用 (CORS) 問題。
4. **系統測試與 Bug 修正**：開發完成後，全組針對功能性、防禦性與響應式排版進行整合測試，修復了長文字跑版、無效金額寫入等 Bug，最後完成最佳化。

## 2. 遇到的問題與解決方法 (Issues & Solutions)

### 問題一：歷史紀錄中，備註輸入長字串或連續字元時，會擠壓金額並溢出卡片
* **原因分析**：`.item-note` 預設未設定折行，當輸入連續英數字或極長備註時，Flex 佈局會自動將其視為一整行，從而把右側的金額欄位推出卡片外。
* **解決方法**：
  在 CSS 中，對包含備註的 `.item-left` 容器設定 `min-width: 0;` 允許其收縮；在 `.item-note` 中加入 `word-break: break-all;` 與 `overflow-wrap: break-word;` 強制折行；同時，為金額容器 `.item-right` 加上 `flex-shrink: 0;` 確保其寬度不被壓縮。

### 問題二：手機版（寬度 480px 以下）儀表板的三個統計卡片太窄，導致文字與圖標擠壓重疊
* **原因分析**：原本在行動裝置上仍維持三欄式網格排版 (`repeat(3, 1fr)`)，但在螢幕寬度不足時，數值（例如：+$10,000）會因為空間不足而與圖標及文字標籤重疊。
* **解決方法**：
  重構 CSS 媒體查詢，在螢幕寬度小於 480px 時，將 `.summary-grid` 的網格改為單欄垂直堆疊 (`grid-template-columns: 1fr;`)。並將各卡片內部改為水平 flex 佈局 (`flex-direction: row; justify-content: space-between;`)，使圖標在左、文字數值在右，兼顧美觀與空間利用率。

### 問題三：金額欄位缺乏邊界限制，使用者可輸入 0 或負數金額
* **原因分析**：前後端在早期串接時，未對傳入的 `amount` 值進行數值邊界判定，導致資料庫中可能寫入無效或負數的交易，造成結餘計算邏輯錯亂。
* **解決方法**：
  1. **前端攔截**：在 JS 表單提交事件中，將輸入值轉換為浮點數，判斷 `isNaN(amount) || amount <= 0`。若成立，則利用 `showToast` 顯示紅色錯誤提示，阻擋其送出。
  2. **後端防禦**：在 Flask `add_expense` API 中，利用 `try-except` 將 `amount` 轉換為 float，並檢驗是否大於 0。若不大於 0，立即回傳 HTTP 400 錯誤與 JSON 錯誤訊息，實作雙重防禦。

### 問題四：Flask 執行目錄非專案目錄時，讀取 index.html 報 404 錯誤
* **原因分析**：`app.py` 中使用 `send_from_directory('.', 'index.html')`，當在其他資料夾路徑下執行該 Python 檔案時，相對路徑 `.` 會指向執行指令的目錄，而非 Python 檔案本身的目錄。
* **解決方法**：
  在 `app.py` 中使用絕對路徑。透過 `os.path.abspath(__file__)` 取得當前檔案的絕對路徑，並解析出其所在的資料夾 `BASE_DIR`。資料庫檔案 `DB_FILE` 與靜態資源目錄皆以此絕對路徑進行拼接，徹底解決路徑定位錯誤。

## 3. 專題成果 (Project Outcomes)

* **極簡高質感的網頁介面**：採用精心設計的漸層背景（深邃湛藍到紫羅蘭色），並搭配卡片式的毛玻璃半透明質感 (Glassmorphism)，使視覺感受極為現代與 premium。
* **流暢的動態回饋 (Micro-animations & Toasts)**：當新增記帳成功時，畫面會滑入精緻的綠色 Toast 提示，失敗則滑入紅色 Toast。按鈕具有 Hover 縮放與漸變效果。
* **即時統計儀表板**：使用者新增任何一筆交易後，無需重新整理頁面，上方的「總收入」、「總支出」及「目前結餘」卡片會即時變動，且金額自動格式化為帶有正負號與千分位逗號（例如：`+$12,500`）。
* **健全的資料庫存取**：所有數據皆即時寫入 SQLite 資料庫中，關閉瀏覽器或重啟伺服器後，歷史紀錄依然完整保存。

## 4. 未來可擴充功能與發展建議 (Future Scope)

為使本系統從一個「單機極簡記帳」演進為「商用/實用級財務管理工具」，我們提出以下五點未來擴充規劃：

1. **使用者註冊與登入系統 (User Authentication)**
   * **說明**：目前為單一使用者版本。未來將引進 JWT (JSON Web Tokens) 驗證機制與加密技術 (如 bcrypt)，實作使用者註冊、登入與登出功能，使每位使用者皆能安全地管理各自的私密帳本。
2. **多維度資料視覺化圖表 (Data Visualization)**
   * **說明**：利用 Chart.js 或 ECharts 等前端圖表庫，將歷史記帳數據轉化為動態圖表。例如：「當月消費分類圓餅圖」（一目了然伙食、交通、娛樂占比）與「年度收支走勢折線圖」（分析儲蓄進度）。
3. **多格式資料匯出功能 (Data Export)**
   * **說明**：在歷史清單旁新增「匯出」按鈕，後端利用 `pandas` 或 `openpyxl` 程式庫，支援使用者將記帳歷史一鍵下載為 Excel (.xlsx) 或 CSV 檔，利於個人做更進一步的資料整理或報稅使用。
4. **雲端資料庫升級 (Cloud Database Migration)**
   * **說明**：將本地端的 SQLite 資料庫，遷移至雲端關聯式資料庫託管服務（如 Supabase, PostgreSQL 或 Firebase）。如此一來，使用者便能在多台裝置（電腦、平板、手機）上即時同步記帳資料。
5. **行動端 PWA 支援與手機 App 打包 (Mobile App & PWA)**
   * **說明**：將現有的響應式網頁配置 Web App Manifest 與 Service Worker，使其升級為「漸進式網頁應用 (PWA)」，讓使用者能將網頁以 App 圖示安裝至手機主畫面，並支援離線記帳。未來亦可使用 Capacitor 或 React Native 將其打包發佈至 App Store 與 Google Play。
